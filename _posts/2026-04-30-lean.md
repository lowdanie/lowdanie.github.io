# Introduction

[Lean](https://lean-lang.org/) is a [functional](https://lean-lang.org/functional_programming_in_lean)
programming language whose type system is powerful enough to encode mathematical proofs.
One implication is that it is possible to use `lean` both to write programs and to
construct mathematical proofs about a programs properties.

The goal of this post is to introduce `lean` by using it to implement insertion sort 
and prove that the output is always sorted.

As a warm up, we'll define the `LessThan` relation on natural numbers and prove basic 
properties such as transitivity.

Next we'll define the `IsLowerBound` and `IsSorted` list properties which are needed
to state and prove properties of insertion sort.

Finally, we'll implement insertion sort and prove that its output is always sorted.

# Dependent Types

Lean bridges the gap between programs and proofs via
[dependent types](https://lean-lang.org/functional_programming_in_lean/Programming-with-Dependent-Types/#Functional-Programming-in-Lean--Programming-with-Dependent-Types).

To see how this works, we'll use dependent types to define a `LessThan` relation
on natural numbers and prove some of its properties.

## LessThan

In this section we'll introduce the `LessThan` dependent type and give a high level 
overview of the relationship between dependent types and proofs.

The specifics of the `lean` syntax will be provided in the following section.

The `LessThan` type represents the "less than" relation between pairs of natural numbers.
It is defined as follows:

```lean
inductive LessThan : Nat → Nat → Prop where
  | base : (n : Nat) → LessThan n (n + 1)
  | step : (n : Nat) → (m : Nat) → LessThan n m → LessThan n (m + 1)
```

At a high level, this means that `LessThan` defines a _family_ of `Prop` types parameterized by
pairs of natural numbers. Objects of types in the family can be built recursively 
via the `base` and `step` constructors.

For example, we can use the `base` constructor to build an element of type `LessThan 0 1`:

```lean
LessThan.base 0 : LessThan 0 1
```

We can then use the `step` constructor to build an element of type `LessThan 0 2`:

```lean
LessThan.step 0 1 (LessThan.base 0) : LessThan 0 2
```

We call `LessThan` a _dependent type_ because each pair of natural numbers `n m : Nat`
defines a different type `LessThan n m`.

The `Prop` in the definition means that types in this family (such as `LessThan 0 1`)
belong to special class of types called _propositions_. In `lean` a proposition type is
defined to be _true_ if there exists an object of that type and _false_ if not.
An object in a `Prop` type is called a _proof_ of the proposition.

For example, `LessThan 0 1` is a proposition and we can write this explicitly as

```lean
LessThan 0 1 : Prop
```

Furthermore, this proposition is true and the proof is the object `LessThan.base 0`
we constructed above:

```lean
LessThan.base 0 : LessThan 0 1
```

We can declare the relationship between the proposition and the proof more explicitly
using the `theorem` keyword:

```lean
theorem zero_lt_one : LessThan 0 1 := LessThan.base 0
```

As an example of a false proposition, consider

```lean
LessThan 0 0 : Prop
```

Indeed, it is easy to see that neither of
the available constructors of `LessThan` can output an object of type `LessThan 0 0`.
For example the `LessThan.base` constructor always outputs objects of type `LessThan n n + 1`
for some natural number `n : Nat`.
And there does not exist a natural number `n` such that `n = n + 1 = 0`.

## Currying

In `lean`, if `S` and `T` are types then
the expression `S → T` is the type representing functions from `S` to `T`.
The arrow `→` is associative to the right. For example: 

```lean
Nat → Nat → Nat = Nat → (Nat → Nat)
```

In other words, `Nat → Nat → Nat` represents functions `f` that take a natural number
`n : Nat` and return a _function_ 

```lean
f(n) : Nat → Nat
```

This means that we can use `f` to map a pair of natural numbers `n m : Nat` to an
output in `Nat` via:

```lean
f(n)(m) : Nat
```

The process of chaining together functions of a single variable to represent multi-variable
functions is called
[currying](https://en.wikipedia.org/wiki/Currying).

The upshot is that we can think of types such as

```lean
S → T → U
```

As representing functions that take two arguments with types `S` and `T` and output
an object of type `U`.

## Syntax

We can now explain the syntax of our `LessThan` definition in more detail. Recall
that `LessThan` was defined by:

```lean
inductive LessThan : Nat → Nat → Prop where
  | base : (n : Nat) → LessThan n (n + 1)
  | step : (n : Nat) → (m : Nat) → LessThan n m → LessThan n (m + 1)
```

The first line says that `LessThan` is an inductive dependent type that is in the `Prop`
universe and is indexed by two natural numbers. Specifically,
`Nat → Nat → Prop` represents functions from pairs of natural numbers to propositions.

The next two rows define the constructors.

First consider the `base` constructor:

```lean
base : (n : Nat) → LessThan n (n + 1)
```

This means that the `base` constructor takes a natural number `n : Nat` and returns
an object of type `LessThan n (n + 1)`. In the language of proofs, we can think of this
as an axiom saying that for all natural numbers `n : Nat`, `n` is less than `n + 1`.

Now consider the `step` constructor:

```lean
step : (n : Nat) → (m : Nat) → LessThan n m → LessThan n (m + 1)
```

This constructor takes three arguments as input - two natural numbers `n m : Nat`
and an object of type `LessThan n m`. It returns an object of type `LessThan n (m + 1)`.
In the language of proofs, this axiom states that for all natural numbers `n m : Nat`,
if `n` is less than `m`, then `n` is less than `m + 1`.

The point of these definitions is that by combining the `base` and `step` 
constructors we can construct objects of type
`LessThan n m` for all `n m : Nat` where `n` is less than `m`.

For example:

```lean
LessThan.base 1 : LessThan 1 2
LessThan.step 1 2 (LessThan.base 1) : LessThan 1 3
LessThan.step 1 3 (LessThan.step 1 2 (LessThan.base 1)) : LessThan 1 4
```

In other words, `LessThan` is an inductive definition of the
 "less than" property on natural numbers.

## Zero Is Less Than Successors

We'll now prove some properties of `LessThan`.
We've already proved some properties that refer to specific numbers. For example:

```lean
theorem one_lt_two : LessThan 1 2 := .base 1
```

The this theorem states the property `LessThan 1 2` and proves it by constructing
and object in `LessThan 1 2` via the `base` constructor.

We can use `one_lt_two` to prove another theorem:

```lean
theorem one_lt_three : LessThan 1 3 := .step 1 2 one_lt_two
```

Now let's prove some more general theorems.

We'll start by proving that for all natural numbers `n : Nat`, `0` is less than
`n + 1`. Intuitively we can prove this by induction on `n`. For the base case,
we can use `LessThan.base 0` to construct `LessThan 0 1`. For the inductive step,
suppose that we have managed to construct an object in `LessThan 0 n` for some
natural number `n : Nat`. We can then use `Lean.step 0 n (LessThan 0 n)` to
construct `LessThan 0 (n + 1)`.

Here is how we can express this in `lean`, followed by more details: 

```lean
theorem LessThan.zero_lt_succ : (n : Nat) → LessThan 0 (n + 1)
  -- case 1: n = 0
  -- need to construct: LessThan 0 (0 + 1)
  | 0 => .base 0
  -- case 2: n = n_prev + 1.
  -- need to construct: LessThan 0 (n_prev + 1 + 1)
  -- proof: recursively call zero_lt_succ n_prev to obtain:
  --        LessThan 0 (n_prev + 1).
  --        then perform one step to obtain: LessThan 0 ((n_prev + 1) + 1).
  | n_prev + 1 => .step 0 (n_prev + 1) (.zero_lt_succ n_prev)
```

The name `LessThan.zero_lt_succ` simply puts `zero_lt_succ` in the `LessThan` namespace.

Note that the type of `LessThan.zero_lt_succ`, `(n : Nat) → LessThan 0 (n + 1)`,
represents _functions_ from `(n : Nat)` to `LessThan 0 (n + 1)`. This means that to
construct an object of type `LessThan.zero_lt_succ` we need to define a function that
takes an arbitrary natural number `n : Nat` and returns an object of type
`LessThan 0 (n + 1)`. In the language of proofs, this means that for all natural numbers
`n : Nat` we must prove that `0` is less than `n + 1`.

The theorem then uses `lean`'s _matching_ syntax to define a function of type
`(n : Nat) → LessThan 0 (n + 1)`. This syntax matches the input `n : Nat` to the available
`Nat` constructors. The natural numbers type `Nat` is itself an inductive type with two
constructors: `zero` which constructs `0` and `succ` which takes an argument `n : Nat`
and returns `n + 1 : Nat`.

The line

`| 0 => .base 0`

matches the case where `n` was constructed via `Nat.zero`. I.e `n=0`. In this case
we need to construct an object of type `LessThan 0 1` which we can do via the `LessThan.base`
constructor. Since the output type is `LessThan`, `lean` lets us write this succinctly as
`.base 0`.

The line

`| n_prev + 1 => .step 0 (n_prev + 1) (.zero_lt_succ n_prev)`

matches the case where `n` was constructed via the `Nat.succ` constructor via
`n = Nat.succ n_prev` for some `n_prev : Nat`. Equivalently, `n = n_prev + 1`.
In this case we need to construct an object of type `LessThan 0 (n_prev + 1)`.
We can do this using the `LessThan.step` constructor.

Since we are calling the `LessThan.step` constructor with arguments `0` and `n_prev + 1`,
we also need to pass it an object of type `LessThan 0 (n_prev + 1)`. We can build
this object by recursively calling `LessThan.zero_lt_succ n_prev`. During compilation,
`lean` verifies that the recursion terminates. This essentially works by by checking that
in the body of `LessThan.zero_lt_succ n`, we only make recursive calls with inputs to
a constructor of `n`. In this case, we are calling `LessThan.zero_lt_succ n_prev`
and `n = Nat.succ n_prev`. We would get a compilation error of for example we tried to
recursively call `LessThan.zero_lt_succ n`.

Rather than using recursion, `lean` also provides a _tactics_ syntax that we can use
to write the same proof in a style that resembles a traditional math proof:

```lean
theorem LessThan.zero_lt_succ (n : Nat) : LessThan 0 (n + 1) := by
  induction n with
  -- base case: n = 0. need to construct: LessThan 0 (0 + 1)
  | zero => exact .base 0
  -- inductive step:
  -- n = n_prev + 1.
  -- inductive hypothesis ih = LessThan 0 (n_prev + 1).
  -- need to construct: LessThan 0 ((n_prev + 1) + 1)
  | succ n_prev ih => exact .step 0 (n_prev + 1) ih
```

The first difference is that, instead of currying,
we explicitly put `LessThan.zero_lt_succ`'s argument
`n : Nat` before the colon. Then we use the `by` keyword to enter into _tactic mode_.  

In tactic mode we can use the `induction` tactic to split the goal into two cases -
the base case where `n=0` and an inductive step where `n=n_prev+1` and we are given
an inductive hypothesis 

```lean
ih : LessThan 0 (n_prev + 1)
```

The main difference from the recursive version is that, rather than constructing
`LessThan 0 (n_prev + 1)` explicitly by a recursive call it is provided to us as
the inductive hypothesis `ih`.
